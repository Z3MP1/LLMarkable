# Overview  
The Document Conversion Pipeline is a Python-based tool that transforms source files (PDF, HTML, etc.) into high-quality, LLM-friendly Markdown chunks. It addresses the common problem of poorly structured document conversion that leads to information loss and suboptimal chunking for LLM consumption. This tool is designed for developers and data scientists who need to prepare documents for retrieval-augmented generation (RAG) systems.

# Core Features  
## Intelligent Chunk Management
- **Token-based chunking**: Uses precise token counting (not character-based) for optimal chunk sizing
- **Smart consolidation**: Merges small chunks with neighbors to prevent information loss
- **Configurable parameters**: chunk_size (2048 tokens), min_tokens (200), chunk_overlap (100)

## Format-Specific Pipelines
- **PDF Pipeline**: Optimized for PDF documents with table structure preservation
- **HTML Pipeline**: Maintains document structure and formatting during conversion
- **Extensible Architecture**: Abstract base class allows easy addition of new format handlers

## Research-Backed Configuration
- Parameters based on industry best practices from Unstructured.io, Pinecone, and Docling documentation
- Token-based approach aligns with embedding model constraints
- Avoids redundant parameters and over-engineering

# User Experience  
## User Personas
- **Data Scientists**: Need clean, well-structured documents for RAG systems
- **Developers**: Require reliable document conversion for their applications
- **Researchers**: Want to preserve all information while making it LLM-friendly

## Key User Flows
1. **Basic Conversion**: User provides PDF/HTML file → System converts to markdown chunks
2. **Batch Processing**: User processes multiple files → System handles each with appropriate pipeline
3. **Configuration Override**: User adjusts chunk parameters → System applies custom settings

## UI/UX Considerations
- Clear CLI interface with progress indicators
- Informative error messages for troubleshooting
- Simple configuration with sensible defaults

# Technical Architecture  
## System Components
### Core Package Structure
```
converter/
├── __init__.py
├── config.py           # Dataclass-based configuration
├── pipelines/
│   ├── __init__.py
│   ├── base.py        # Abstract base class (COMPLETE)
│   ├── pdf.py         # PDF-specific pipeline
│   └── html.py        # HTML-specific pipeline
├── utils.py           # Chunk consolidation utilities
└── main.py            # CLI entry point
```

### Key Classes
- **Config**: Simple dataclass with chunk_size, min_tokens, chunk_overlap
- **BasePipeline**: Abstract class defining process() and supported_extensions
- **PDFPipeline/HTMLPipeline**: Format-specific implementations
- **CLI**: Typer-based command-line interface

## Data Models
- **Configuration**: Token-based parameters (no character-based filtering)
- **Chunks**: List of markdown strings with metadata
- **Pipeline Interface**: Input Path → List[str] output

## APIs and Integrations
- **Docling**: Core document conversion library
- **Transformers**: HuggingFace tokenizers for precise token counting
- **Rich**: Console output and progress tracking
- **Typer**: CLI framework

## Infrastructure Requirements
- Python 3.12+
- uv package manager
- Minimal dependencies (docling, transformers, typer, rich)

# Risks and Mitigations  
## Technical Challenges
- **Risk**: Complex PDF structures may fail conversion
- **Mitigation**: Fallback to HierarchicalChunker, save as single file if needed

- **Risk**: Token counting performance with large documents
- **Mitigation**: Cache tokenizer, process in streaming fashion

## Scope Creep
- **Risk**: Adding Phase 2 features to Phase 1
- **Mitigation**: Strict MVP discipline, clear phase separation

## Over-Engineering
- **Risk**: Creating enterprise-level complexity for simple tool
- **Mitigation**: Regular sense-checks, prefer simple solutions

# Appendix

- **Chunking Best Practices**: Token-based preferred, 200-2048 token range optimal
- **Docling Patterns**: HybridChunker with HierarchicalChunker fallback
